# AI 游戏数字商品客服与工单调度平台总体设计

## 1. 文档目的

本文档固化项目的开发目标、系统边界、总体架构、实施阶段、质量门禁及 Skill/Plugin 使用策略。项目已经从“泛电商售后客服”调整为“游戏数字商品商城与 AI 客服工单平台”。后续详细需求、ADR、数据模型和接口设计必须与本文档保持一致。

## 2. 项目目标

构建一个可写入简历、能够完整演示、具备企业级工程特征的个人项目。第一版以作品级 MVP 为目标，优先完成稳定的游戏数字商品交易咨询、客服接待、管理员工单处理和角色权限控制闭环；系统在模块边界、权限、审计、数据隔离、测试、部署和可观测性方面预留生产级演进路径。

第一版不做淘宝/拼多多/京东式多商户平台，而是做单平台自营的类 Steam/Epic/PlayStation Store/Xbox Store 游戏数字商品商城。MVP 使用虚拟演示数据，不接入真实支付、真实退款、真实平台 API 或真实 CDK 供应链。

## 3. 已确认决策

1. 交付策略：先完成作品级 MVP，再按实际需要扩展为生产级系统。
2. 业务域：游戏、DLC、CDK、礼物码、数字权益、退款审核和账号处置。
3. 平台模式：单平台自营；不做多商户入驻。底层保留 `tenant_id` 作为企业级隔离和未来扩展能力。
4. 后端形态：Spring Boot 模块化单体，Python Agent 作为独立服务。
5. 前端形态：`customer-web` 和 `staff-web` 两个 React 应用，共享组件、类型和基础工具包。
6. B 端角色：客服、管理员、超级管理员。客服负责对话和升级；管理员处理业务工单；超级管理员配置角色模块和系统能力。
7. AI 接入：Java 先定义稳定的 Agent 契约并提供 Mock 实现；真实 Python Agent 在核心前端与 Java 业务完成后开发。
8. 模型策略：通过 OpenAI-compatible 适配层接入云端模型，同时提供 Mock 模型保证无密钥环境可测试和演示。
9. 推进方式：采用业务纵向切片，每个阶段必须交付可运行、可测试的业务增量；真实 Agent 通过 Mock 契约延后接入。
10. 部署策略：Windows 主机承担代码开发和调试，Linux 虚拟机承担中间件和集成部署。

## 4. 系统范围

### 4.1 第一版包含

- C 端游戏商城首页、分类、搜索、游戏卡片、游戏详情、购物车入口、个人入口和客服入口。
- 演示游戏、DLC、CDK、礼物码、订单和数字权益数据。
- 消费者围绕订单、CDK、礼物、退款和账号问题发起咨询。
- 客服登录、接待队列、实时会话、用户与订单/权益上下文、Agent 建议、知识库命中和升级工单。
- 管理员查看所有客服会话、管理客服、处理业务工单、退款审核、CDK/礼物补发、账号冻结/解冻/注销审核。
- 超级管理员配置角色可见模块、权限策略、SLA、知识库、Agent 配置和审计。
- Mock Agent，以及后续真实 Agent 的意图识别、RAG、业务工具调用、自动建单建议、摘要和推荐回复。
- 操作审计、错误追踪、日志、指标、健康检查、基础压测、备份和恢复演练。
- Docker Compose 分组部署、演示种子数据、部署手册和作品演示材料。

### 4.2 第一版不包含

- 真实支付、真实退款打款、真实 Steam/Epic/PlayStation/Xbox API。
- 多商户入驻、商家后台、平台抽佣、分账结算和商家客服体系。
- 真实 CDK 采购、真实库存资金风控和真实用户资产迁移。
- Kubernetes、多节点数据库、跨地域容灾和真实生产高可用承诺。
- 本地大模型训练或在 RTX 4060 Laptop GPU 上长期托管大模型。

## 5. 总体架构

### 5.1 前端

- `frontend/customer-web`：游戏商城、分类搜索、游戏详情、订单/权益入口、客服咨询和工单进度。
- `frontend/staff-web`：客服工作台、管理员工单中心、退款/账号处置、客服管理、角色模块配置和数据看板。
- `frontend/packages/ui`：共享设计令牌和无业务状态的通用组件。
- `frontend/packages/api-client`：由 OpenAPI 契约生成或约束的类型安全客户端。
- `frontend/packages/shared`：通用类型、校验规则和无框架业务常量。

两个应用保持独立路由、权限和构建产物，避免 C 端商城导航与 B 端权限逻辑互相污染。

### 5.2 Java 模块化单体

Java 应用按业务能力拆分模块：

- identity：认证、用户、角色、模块权限、员工、账号状态和租户上下文。
- catalog：Steam 游戏、游戏版本、购买方式、可售方案、分类、媒体资源和商品知识。
- order：订单、订单项和演示支付快照。
- fulfillment：数字权益、CDK、礼物、交付异常和补发。
- conversation：会话、消息、客服队列和在线状态。
- ticket：工单、分类、状态机、SLA、退款审核、补发记录和账号处置记录。
- knowledge：知识库元数据、文档任务和检索接口。
- notification：站内通知和异步消息适配。
- analytics：业务事件聚合及看板查询。
- agent-gateway：Agent 调用契约、Mock 实现、远程实现和降级策略。

模块只通过公开应用服务、领域事件或明确的查询接口协作。数据库可以共用 MySQL 实例，但表归属和代码依赖必须保持单向、可验证。

### 5.3 Agent 服务

真实 Agent 是独立 FastAPI 服务，通过版本化 HTTP API 与 Java 通信。Agent 不直接访问核心业务数据库，只能调用 Java 暴露的受控工具接口。该边界确保权限、审计、事务和业务规则仍由 Java 负责。

Agent 采用可组合工作流：输入校验、意图识别、上下文获取、知识检索、工具选择、响应生成、转人工判断和结果审计。退款、补发和账号处置等高风险动作只能生成建议或草稿，必须由授权管理员确认。

## 6. 核心数据流

### 6.1 游戏购买/权益咨询

消费者通过 `customer-web` 浏览游戏并进入详情或订单/权益入口，创建会话。Java 校验身份和资源归属，调用 `AgentGateway`。Agent 未接入时由 Mock 返回确定性结果；Agent 接入后由远程适配器调用 Python 服务。Java 负责消息落库、审计、超时、降级和向前端推送。

### 6.2 转客服与业务工单

Agent 或消费者触发转人工后，会话进入客服队列。客服接入动作必须使用并发控制，确保同一会话不会被多名客服同时占用。客服无法直接处理退款、补发或账号处置，只能升级业务工单。管理员在工单中心处理具体业务动作。

### 6.3 角色模块控制

超级管理员配置角色可见模块。B 端加载菜单时，Java 根据用户角色、模块授权和资源权限返回模块列表。前端只展示授权模块；后端仍对每个动作执行 RBAC、资源归属、状态机、幂等和审计校验。

### 6.4 知识库

管理员上传文档至 MinIO，Java 记录文档元数据并发布解析任务。解析消费者提取、分块和索引内容；检索接口返回带来源标识的结果。真实 Agent 只能基于检索结果生成答案，并将引用来源、提示版本、模型信息和耗时写入调用日志。

## 7. 错误处理与降级

- 所有外部接口使用统一错误结构、关联 ID 和可稳定识别的业务错误码。
- 前端区分校验错误、业务冲突、权限不足、暂时不可用和未知错误。
- Agent 超时、限流或不可用时，Java 返回降级答复并提供转人工入口，不能阻塞核心客服流程。
- 退款、补发、账号处置失败不得伪造成功状态，必须保留工单和审计。
- MQ 消费者采用幂等键、有限重试和死信队列；无法自动恢复的消息进入人工处理流程。
- 工单状态变更使用显式状态机和乐观并发控制，非法跳转必须被拒绝并记录。

## 8. 安全与权限

- 采用最小权限 RBAC，并通过模块权限控制 B 端菜单与页面入口。
- 消费者只能访问本人订单、权益、会话和对外工单进度。
- 客服只能对话和升级工单，不能执行退款、补发或账号处置。
- 管理员执行高风险动作必须二次确认、填写原因并写审计。
- 超级管理员配置角色模块时必须保留系统可恢复路径，不能移除最后一个系统管理员能力。
- Agent 不持有数据库凭据，也不能绕过 Java 的授权和业务校验。
- API 密钥、数据库密码和令牌只通过环境变量或密钥文件注入，示例配置不包含真实凭据。
- CDK 明文和账号敏感字段不得进入普通日志。

## 9. 开发阶段

阶段编号保持不变：

1. 阶段 0：需求与架构基线。
2. 阶段 1：工程与环境基线。
3. 阶段 2：体验、契约与 Mock。
4. 阶段 3：消费者业务闭环。

6. 阶段 5：知识库与工程增强。

阶段内容按新游戏数字商品业务域执行，但阶段节奏和质量门禁不变。

## 10. 项目成功标准

- 无真实模型密钥时，Mock Agent 仍可完整演示消费者游戏购买问题到工单关闭的业务闭环。
- 接入真实 Agent 后，前端和核心 Java 业务不需要改变调用契约。
- C 端页面具备游戏商城的视觉质量和基本信息架构，不再像普通售后后台。
- B 端能演示客服、管理员、超级管理员三类角色的模块显示差异和业务职责差异。
- 两个前端应用、Java 应用和基础中间件可由文档化命令构建、测试和启动。
- 关键业务规则、权限、状态迁移和并发冲突都有自动化测试证据。
- 全新环境可使用 Compose 和种子数据复现演示系统。

# Stage 2 Route Map

## Purpose

本文档作为阶段 2 的页面路由与信息架构基线。后续前端页面原型、OpenAPI 契约、Java Mock API、权限矩阵和演示数据都以本文档为准。

当前项目重点围绕游戏数字商品下单、订单权益、客服会话和工单处理闭环，不在第一版展开完整多商户平台、复杂运营活动或真实支付清结算。

## Customer Web Search Flow

搜索页是 C 端必要页面，不应只停留在首页输入框。

核心交互：

1. 用户在首页顶部搜索框输入游戏名称或关键词。
2. 跳转到 `/search?q=关键词`。
3. 搜索页展示结果列表、排序、价格区间、平台、交付方式、折扣筛选。
4. 点击游戏卡片进入 `/games/:id`。

阶段 2 只实现静态或 Mock 搜索结果，不接入真实搜索引擎。

### Staff route grouping

B 端会话和工单都采用“列表分组 + 详情页”的结构：

```text
客服会话
├─ 待接待列表：/conversations/pending
├─ 已接待列表：/conversations/accepted
└─ 会话详情：/conversations/:id

工单中心
├─ 待处理列表：/tickets/pending
├─ 已处理列表：/tickets/resolved
└─ 工单详情：/tickets/:id
```

这样做的原因：

- 待办和历史天然是不同工作模式，混在一个列表会降低客服和管理员效率。
- 详情页必须保留，因为会话和工单都需要展示上下文、操作记录、审计和状态流转。
- URL 可直接表达当前模块，便于刷新、收藏、权限拦截和后续菜单高亮。

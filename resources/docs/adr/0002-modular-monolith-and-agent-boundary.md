# ADR-0002：模块化单体与独立 Agent 边界

- 状态：Accepted
- 日期：2026-06-22

## 背景

售后平台具有身份、订单、售后、会话、工单、知识、通知和统计等业务能力。完整微服务会放大个人项目的部署、调试、契约和数据一致性成本；Agent 又需要 Python 生态、模型依赖和独立资源治理。

## 决策

- Java 使用 Spring Boot 模块化单体承载全部核心业务和事务。
- Python Agent 作为独立 FastAPI 服务。
- Java 通过 AgentGateway 端口提供 Mock 和远程两种适配器。
- Agent 不直连核心数据库，只调用 Java 提供的受控工具。
- 前端和 Java 业务先完成，真实 Agent 在第 6 阶段接入。

## 理由

- 模块化单体保留清晰领域边界，同时降低分布式事务和运维复杂度。
- 独立 Agent 隔离模型依赖、失败、超时和 Python 运行时。
- Mock Agent 使无模型密钥环境可测试，防止 AI 开发阻塞核心业务。
- Java 继续作为授权、业务规则、事务和审计的唯一执行边界。

## 后果

- Java 模块依赖必须用 Spring Modulith 或 ArchUnit 自动验证。
- Agent 契约需要版本化、超时、降级、契约测试和调用审计。
- 模块共享一个 MySQL 实例，但表和 Repository 所有权不得跨模块泄漏。
- 未来拆服务应由独立伸缩、发布或组织边界驱动，而不是为了展示技术数量。

## 重新评估条件

某模块出现独立团队、独立伸缩、独立数据合规或频繁发布冲突，并且监控数据证明拆分收益高于分布式成本时，新增拆分 ADR。

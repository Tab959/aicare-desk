# 产品简述

## 1. 产品定位

本项目从“泛电商售后客服”调整为“游戏数字商品商城与 AI 客服工单平台”。C 端是一个类 Steam/Epic/PlayStation Store/Xbox Store 的自营游戏数字商品平台，销售演示游戏、DLC、游戏 CDK、礼物码和点卡类权益；B 端是客服、管理员和超级管理员使用的客服会话、业务工单、退款、账号处置、权限配置与运营看板系统。

第一版不是淘宝、拼多多、京东式多商户平台，不做商家入驻和多店铺治理。系统按单平台/单企业运行，但底层仍保留 `tenant_id`，用于工程上体现企业级隔离能力和后续扩展空间。

## 2. 要解决的问题

游戏数字商品售后和普通实物电商不同，核心问题集中在“数字权益交付”和“账号风险”：

- 用户购买后没有收到 CDK、礼物链接或游戏权益。
- CDK 已被兑换、兑换失败、区域不匹配或平台不匹配。
- 礼物赠送失败、收礼人未领取、过期或重复领取。
- 用户要求退款，但数字商品可能已经激活，需要按平台规则审核。
- 用户账号存在异常登录、误封、违规、注销或资产争议。
- 客服需要查看用户、订单、CDK、礼物、会话和历史工单上下文，但不能直接执行高风险动作。
- 管理员需要处理退款、账号处置和业务工单，并能查看客服对话质量。
- 超级管理员需要配置角色能看到哪些系统模块，体现 RBAC/模块权限控制能力。

## 3. 用户与角色

- 消费者：浏览游戏、搜索分类、查看游戏详情、购买演示数字商品、接收 CDK/礼物、发起客服咨询、查看工单进度、确认结果并评价。
- 客服：负责与用户对话，查看必要上下文，使用 Agent 建议和知识库回复用户，必要时升级为业务工单；不直接退款，不直接冻结/注销账号。
- 管理员：管理客服，查看所有客服会话，处理业务工单，执行受控退款、CDK 补发、礼物重发、账号冻结/解冻/注销审核等高风险业务动作。
- 超级管理员：拥有管理员能力，并能配置角色、模块可见性、权限策略、SLA、知识库、审计和系统参数。
- Agent：在 Java 授权边界内识别意图、生成答复、推荐工单分类、生成摘要和建议处理动作；不能直接修改订单、账号、退款或工单状态。

## 4. 核心业务结果

消费者的问题最终必须落到以下结果之一：

1. Mock Agent 或真实 Agent 基于订单、权益、知识库给出确定性答复；
2. 会话转入客服接待，由客服继续沟通；
3. 客服升级为工单，由管理员处理退款、补发、账号处置或规则解释；
4. 处理结果同步给消费者，消费者确认并评价。

## 5. MVP 边界

第一版包含：

- 游戏商城首页、分类导航、搜索、游戏卡片、游戏详情、购物车入口和个人入口的高保真原型。
- 演示游戏、DLC、CDK、礼物码、订单和权益数据。
- 用户围绕订单、CDK、礼物、退款、账号问题发起咨询。
- Mock Agent 识别常见游戏售后意图并给出建议。
- 客服接待会话、查看用户上下文、采纳 Agent 建议、升级工单。
- 管理员处理工单、退款审核、CDK/礼物补发、账号冻结/解冻/注销审核。
- 超级管理员配置角色可见模块，展示 RBAC 与动态菜单能力。

第一版不包含：

- 真实支付、真实退款打款、真实 Steam/Epic/PS/Xbox 平台接口。
- 多商户入驻、商家后台、平台抽佣、结算分账。
- 真实游戏库授权、真实 CDK 采购、真实库存资金风控。
- 本地大模型训练或长期托管大模型。

## 6. 成功指标

- 无模型 API Key 时，Mock Agent 仍能完整演示“游戏购买问题—AI 识别—转客服—升级工单—管理员处理—用户确认评价”。
- 客服、管理员、超级管理员看到的模块不同，且页面能清晰展示角色权限控制。
- 高风险动作全部由管理员或超级管理员确认，且有审计记录。
- 真实 Agent 接入后，前端和 Java 核心业务不改变既有契约。
- 全新环境可根据文档启动基础设施、应用和演示数据。

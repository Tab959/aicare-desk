# 产品简述

## 1. 要解决的问题

电商售后团队需要重复回答订单、物流、退款、商品质量和投诉问题，并在客服、售后、仓库、物流、财务之间人工传递处理信息。传统流程存在响应慢、上下文丢失、责任不清、超时不可见、重复操作和数据难以分析等问题。

本平台将消费者咨询、AI/Mock 自动接待、人工客服、售后申请、内部工单和知识库连接为可追踪的业务闭环。任何无法自动解决的问题都必须能够安全降级到人工处理，任何跨部门任务都必须形成有责任人、有状态、有时限、有记录的工单。

## 2. 用户

- 消费者：查询商品、订单和物流，发起咨询或售后申请，查看进度并评价服务。
- 人工客服：接入会话、查看授权范围内的用户与订单上下文、回复消费者并创建工单。
- 售后人员：审核退货、换货、退款和质量问题。
- 仓库人员：处理补发、换货出库和退件签收。
- 物流人员：处理停滞、丢失、破损等物流异常。
- 财务人员：处理退款确认和发票相关任务。
- 管理员：管理组织、权限、工单规则、知识库、SLA 和统计指标。

## 3. 产品结果

平台必须让消费者的问题最终落到以下结果之一：

1. Mock Agent 或真实 Agent 基于确定性业务数据或知识来源给出答案；
2. 会话转入人工客服队列并由客服处理；
3. 创建内部工单，由指定部门处理并将结果反馈给消费者。

处理过程必须满足身份校验、资源归属、租户隔离、状态机约束、审计追踪和故障降级要求。

## 4. MVP 边界

- 使用内置商品、订单、物流和售后演示数据，不对接真实支付、电商平台或物流平台。
- 按单企业运行，但核心数据、缓存键、权限上下文和审计事件携带租户标识。
- 先实现 Mock Agent；前端与 Java 核心业务不依赖真实模型 API。
- Java 负责业务规则、权限、事务与审计；Python Agent 只能调用 Java 提供的受控工具。
- 本地虚拟机按需运行 MySQL、Redis、RabbitMQ、MinIO、Elasticsearch 和可观测性组件。
- 首版不实现完整 SaaS 开户计费、Kubernetes 集群或本地大模型托管。

## 5. 成功指标

- 没有模型 API Key 时，仍能完整演示“消费者咨询—转人工—创建工单—部门处理—消费者评价”。
- 真实 Agent 接入后，前端和 Java 核心业务不改变既有调用契约。
- 关键权限、状态迁移、并发冲突、幂等和异常路径具有自动化测试证据。
- 全新环境可根据版本化文档启动基础设施、应用和演示数据。
- 简历中的性能、AI 效果和工程质量指标来自可重复运行的测试或评测，而不是估算。

## 6. 明确排除

- 真实资金结算和真实支付回调。
- 完整商城交易、购物车、营销和秒杀。
- 多节点高可用、跨地域容灾和生产容量承诺。
- 未经人工确认的高风险 Agent 操作。

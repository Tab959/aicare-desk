# 核心业务用例

## UC-01 消费者查看订单并发起咨询

**前置条件：** 消费者已登录，订单属于本人和当前租户。

**触发：** 消费者在订单详情选择“咨询此订单”。

**主流程：**

1. Java 校验租户和订单归属。
2. 系统创建或复用该订单的活动会话。
3. customer-web 打开会话并展示脱敏订单摘要。
4. 系统记录会话创建审计和关联 ID。

**替代与失败：**

- 登录过期时返回 401，前端保留当前页面并引导重新登录。
- 订单不属于当前消费者时返回资源不可用结果，不泄露订单是否存在。
- 重复点击使用幂等键，只产生一个活动会话。

**可观察结果：** 返回会话 ID；数据库存在唯一活动会话；消费者只能看到本人订单摘要。

## UC-02 Mock Agent 回答订单问题

**前置条件：** 会话处于 BOT_ACTIVE，订单上下文有效。

**触发：** 消费者发送“我的订单什么时候发货”等确定性问题。

**主流程：**

1. Java 保存消费者消息。
2. agent-gateway 调用 Mock Agent。
3. Mock Agent 根据订单状态返回固定模板答复和问题类型。
4. Java 保存答复并推送给消费者。

**替代与失败：**

- Mock 规则不命中时，返回无法自动解决的标准答复并提供转人工操作。
- AgentGateway 超时或真实 Agent 不可用时，同样走确定性降级，不阻塞会话。
- 重复消息事件通过消息幂等键只入库一次。

**可观察结果：** 会话中出现消费者消息和系统答复；关联 ID 一致；无模型密钥也能完成。

## UC-03 会话转入人工队列

**前置条件：** 会话处于 BOT_ACTIVE，尚未被人工客服接入。

**触发：** 消费者主动转人工，或 Mock/AI 建议转人工。

**主流程：**

1. Java 校验当前会话状态。
2. 会话迁移到 WAITING_HUMAN。
3. 系统创建唯一客服队列项。
4. staff-web 的等待队列收到更新。

**替代与失败：**

- 重复转人工请求返回现有队列状态，不重复建队列项。
- 已关闭会话拒绝转人工并返回 409。
- 推送失败不回滚队列状态，由客户端轮询或重连恢复。

**可观察结果：** 唯一活动队列项、状态变化记录和 HumanHandoffRequested 事件。

## UC-04 客服接入并回复会话

**前置条件：** 客服已登录并具有 conversation:accept；会话正在等待人工。

**触发：** 客服点击“接入”。

**主流程：**

1. Java 以乐观锁或原子操作占用队列项。
2. 会话迁移到 HUMAN_ACTIVE 并绑定客服。
3. staff-web 展示消费者、订单和会话摘要。
4. 客服回复后，消息保存并推送给消费者。

**替代与失败：**

- 两名客服同时接入时，仅一人成功；另一人收到 409 和当前接入人信息。
- 客服失去权限或不在允许范围时返回 403。
- WebSocket 断开后消息仍持久化，重连按游标补拉。

**可观察结果：** 会话只有一个当前客服；双方消息顺序可恢复；接入和回复均有审计。

## UC-05 消费者提交质量问题售后申请

**前置条件：** 订单项属于消费者，处于允许售后的时间窗口。

**触发：** 消费者选择质量问题、填写说明并上传凭证。

**主流程：**

1. 前端校验必填信息、文件类型和大小。
2. Java 再次校验订单归属、售后窗口和文件元数据。
3. 凭证存入租户隔离的 MinIO 路径。
4. 系统创建 SUBMITTED 状态的售后申请。

**替代与失败：**

- 非法文件类型或超限文件返回 400，且不创建售后申请。
- 同一订单项和幂等键重复提交返回原申请。
- 超出售后期返回明确业务错误，不创建对象存储孤儿。

**可观察结果：** 售后申请、凭证引用和审计记录一致；消费者可查看申请编号。

## UC-06 客服创建并分派工单

**前置条件：** 客服正在处理相关会话，业务问题需要部门跟进。

**触发：** 客服确认由 Mock/AI 生成或人工填写的工单草稿。

**主流程：**

1. Java 校验会话权限、工单分类和关联订单/售后申请。
2. 系统创建 UNASSIGNED 工单和首条处理记录。
3. 分派规则选择目标部门，工单迁移到 PENDING。
4. 发布 WorkOrderCreated 和 WorkOrderAssigned 事件。

**替代与失败：**

- 分类无有效处理部门时保留 UNASSIGNED 并通知管理员。
- 重复确认使用幂等键返回原工单。
- Agent 草稿只作为输入，未经客服确认不创建工单。

**可观察结果：** 工单、分派、处理记录和事件使用同一关联 ID。

## UC-07 部门人员处理工单

**前置条件：** 工单分派给当前人员或其部门，状态允许处理。

**触发：** 部门人员开始处理并提交处理结果。

**主流程：**

1. Java 校验角色、租户、分派和当前版本。
2. 工单从 PENDING 迁移到 IN_PROGRESS。
3. 人员登记结果，必要时进入 WAITING_CUSTOMER。
4. 处理完成后迁移到 COMPLETED 并通知消费者。

**替代与失败：**

- 非责任部门返回 403。
- 版本冲突或非法状态跳转返回 409，不产生部分更新。
- 通知失败进入异步重试或死信，不回滚已完成工单。

**可观察结果：** 每次变化都有不可变处理记录、操作者、时间和事件。

## UC-08 消费者确认结果并评价

**前置条件：** 关联工单处于 COMPLETED，消费者拥有对应会话或售后申请。

**触发：** 消费者确认结果并提交 1 至 5 分评价。

**主流程：**

1. Java 校验消费者资源归属和工单状态。
2. 工单迁移到 CLOSED。
3. 会话在无其他活动任务时迁移到 RESOLVED 或 CLOSED。
4. 保存唯一满意度评价并更新统计事件。

**替代与失败：**

- 非本人资源返回资源不可用结果。
- 非 COMPLETED 工单拒绝确认并返回 409。
- 重复评价返回原评价；已提交评价不可覆盖，只允许管理员追加审计备注。

**可观察结果：** 工单关闭、评价持久化、消费者可见最终结果，统计异步更新。

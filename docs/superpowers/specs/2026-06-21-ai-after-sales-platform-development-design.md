# AI 电商售后客服与工单调度平台总体设计

## 1. 文档目的

本文档固化 AI 电商售后客服与工单调度平台的开发目标、系统边界、总体架构、实施阶段、质量门禁及 Skill/Plugin 使用策略，作为后续需求、架构和实施计划的上位约束。

`项目概要.md` 是项目初稿和需求输入；本文档记录已经确认的开发决策。后续详细需求、ADR、数据模型和接口设计必须与本文档保持一致。如需改变已确认决策，应先更新本文档或新增 ADR 说明原因和影响。

## 2. 项目目标

构建一个可写入简历、能够完整演示、具备企业级工程特征的个人项目。第一版以作品级 MVP 为目标，优先完成稳定的电商售后业务闭环；系统在模块边界、权限、审计、数据隔离、测试、部署和可观测性方面预留生产级演进路径。

第一版不追求真实互联网规模、高可用集群或完整 SaaS 商业化能力。企业级价值通过清晰的领域边界、可靠的状态流转、可测试的接口、可观测的运行状态和可复现的部署流程体现。

## 3. 已确认决策

1. 交付策略：先完成作品级 MVP，再按实际需要扩展为生产级系统。
2. 后端形态：Spring Boot 模块化单体，Python Agent 作为独立服务。
3. 前端形态：`customer-web` 和 `staff-web` 两个 React 应用，共享组件、类型和基础工具包。
4. AI 接入：Java 先定义稳定的 Agent 契约并提供 Mock 实现；真实 Python Agent 在核心前端与 Java 业务完成后开发。
5. 模型策略：通过 OpenAI-compatible 适配层接入云端模型，同时提供 Mock 模型保证无密钥环境可测试和演示。
6. 电商底座：内置演示商品、订单、物流和退款数据，不在第一版对接真实电商平台。
7. 租户策略：第一版按单企业运行，但核心业务数据、缓存键、权限上下文和审计事件预留 `tenant_id`。
8. 推进方式：采用业务纵向切片，每个阶段必须交付可运行、可测试的业务增量；真实 Agent 通过 Mock 契约延后接入。
9. 部署策略：Windows 主机承担代码开发和调试，Linux 虚拟机承担中间件和集成部署。

## 4. 系统范围

### 4.1 第一版包含

- 消费者注册、登录、商品、订单、物流、售后申请、咨询、工单进度和服务评价。
- 客服登录、接待队列、实时会话、用户与订单上下文、转人工、会话处理和工单创建。
- 工单分类、优先级、分派、转派、状态机、处理记录、SLA、通知和评价。
- 管理员的用户、角色、部门、客服、工单分类、知识库、SLA 和基础数据看板。
- 文档与 FAQ 知识库、全文检索、异步解析及检索指标。
- Mock Agent，以及后续真实 Agent 的意图识别、RAG、业务工具调用、自动建单、摘要和推荐回复。
- 操作审计、错误追踪、日志、指标、健康检查、基础压测、备份和恢复演练。
- Docker Compose 分组部署、演示种子数据、部署手册和作品演示材料。

### 4.2 第一版不包含

- 真实支付、真实退款打款、真实物流平台或第三方电商平台集成。
- 完整 SaaS 租户开户、套餐、计费、配额和租户自助管理。
- Kubernetes、多节点数据库、跨地域容灾和真实生产高可用承诺。
- 本地大模型训练或在 RTX 4060 Laptop GPU 上长期托管大模型。
- 购物车、营销、秒杀和完整商城交易链路。

## 5. 总体架构

### 5.1 前端

- `apps/customer-web`：消费者商品、订单、咨询、售后、工单进度和评价。
- `apps/staff-web`：客服工作台、工单中心、知识库、组织权限和数据看板。
- `packages/ui`：共享设计令牌和无业务状态的通用组件。
- `packages/api-client`：由 OpenAPI 契约生成或约束的类型安全客户端。
- `packages/shared`：通用类型、校验规则和无框架业务常量。

两个应用保持独立路由、权限和构建产物，避免 C 端与 B 端导航、状态和权限逻辑互相污染。

### 5.2 Java 模块化单体

Java 应用按业务能力而不是传统技术层拆分模块，初始模块包括：

- identity：认证、用户、角色、权限和租户上下文。
- catalog：演示商品及商品知识。
- order：订单、订单项和物流状态。
- aftersales：退款、退货、换货和质量问题申请。
- conversation：会话、消息、客服队列和在线状态。
- ticket：工单、分类、分派、状态机、SLA 和处理记录。
- knowledge：知识库元数据、文档任务和检索接口。
- notification：站内通知和异步消息适配。
- analytics：业务事件聚合及看板查询。
- agent-gateway：Agent 调用契约、Mock 实现、远程实现和降级策略。

模块只通过公开应用服务、领域事件或明确的查询接口协作。数据库可以共用 MySQL 实例，但表归属和代码依赖必须保持单向、可验证。

### 5.3 Agent 服务

真实 Agent 是独立 FastAPI 服务，通过版本化 HTTP API 与 Java 通信。Agent 不直接访问核心业务数据库，只能调用 Java 暴露的受控工具接口。该边界确保权限、审计、事务和业务规则仍由 Java 负责。

Agent 采用可组合工作流：输入校验、意图识别、上下文获取、知识检索、工具选择、响应生成、转人工判断和结果审计。需要暂停或人工确认的高风险动作通过 human-in-the-loop 节点处理。

### 5.4 数据与中间件

- MySQL：核心事务数据和审计数据。
- Redis：缓存、限流、在线状态、短期会话状态和幂等控制。
- MinIO：知识库文档及售后凭证。
- RabbitMQ：文档解析、通知、统计和非关键异步任务。
- Elasticsearch：FAQ、知识文档和历史工单全文检索；向量检索方案在知识库设计阶段通过 ADR 固化。
- Prometheus、Grafana 和日志采集组件：集成验收阶段的可观测性。

中间件只在业务切片需要时启用，禁止为了展示技术栈而引入没有业务消费者的组件。

## 6. 开发与部署拓扑

当前主机为 16 GB 内存环境。虚拟机使用以下轻量配置：

- 4 vCPU。
- 6 GB 内存。
- 4 GB Swap。
- 120 GB 动态扩容磁盘。
- 无桌面的 Linux Server。

前端、Java 和 Python 在 Windows 主机运行以保留 IDE 调试、热更新和快速测试。虚拟机使用 Docker Compose Profiles：

- `core`：MySQL、Redis。
- `messaging`：RabbitMQ、MinIO。
- `search`：Elasticsearch。
- `observe`：Prometheus、Grafana 和日志组件。

日常开发只启动当前切片需要的 Profile。完整容器化应用仅在集成、验收和演示阶段启动。16 GB 主机不长期同时运行全部中间件、应用容器和本地大模型。

## 7. 核心数据流

### 7.1 消费者咨询

消费者通过 `customer-web` 创建会话，Java 校验身份和订单归属，调用 `AgentGateway`。Agent 未接入时由 Mock 返回确定性结果；Agent 接入后由远程适配器调用 Python 服务。Java 负责消息落库、审计、超时、降级和向前端推送。

### 7.2 转人工与工单

Agent 或消费者触发转人工后，会话进入客服队列。客服接入动作必须使用并发控制，确保同一会话不会被多名客服同时占用。需要跨部门处理时，由客服或 Agent 建议创建工单；Java 校验权限与业务规则后创建工单并驱动状态机。

### 7.3 知识库

管理员上传文档至 MinIO，Java 记录文档元数据并发布解析任务。解析消费者提取、分块和索引内容；检索接口返回带来源标识的结果。真实 Agent 只能基于检索结果生成答案，并将引用来源、提示版本、模型信息和耗时写入调用日志。

## 8. 错误处理与降级

- 所有外部接口使用统一错误结构、关联 ID 和可稳定识别的业务错误码。
- 前端区分校验错误、业务冲突、权限不足、暂时不可用和未知错误。
- Agent 超时、限流或不可用时，Java 返回降级答复并提供转人工入口，不能阻塞核心售后流程。
- MQ 消费者采用幂等键、有限重试和死信队列；无法自动恢复的消息进入人工处理流程。
- 工单状态变更使用显式状态机和乐观并发控制，非法跳转必须被拒绝并记录。
- 文档解析、通知和统计失败不得回滚已完成的核心事务，通过最终一致性恢复。

## 9. 安全与权限

- 采用最小权限的 RBAC，并在资源级校验订单归属、会话归属、部门和租户上下文。
- Agent 不持有数据库凭据，也不能绕过 Java 的授权和业务校验。
- API 密钥、数据库密码和令牌只通过环境变量或密钥文件注入，示例配置不包含真实凭据。
- 登录、权限变更、工单变更、知识库发布和 Agent 工具调用写入不可抵赖的审计事件。
- 上传文件限制类型、大小和访问范围，并执行文件名净化和恶意内容防护。
- 日志不得记录密码、完整令牌、模型 API Key 或不必要的个人敏感信息。
- 阶段 0 产出正式威胁模型；阶段 7 根据威胁模型执行安全验证和依赖扫描。

## 10. 测试与质量门禁

所有功能遵循测试驱动的短循环：先写失败测试、最小实现、运行验证、重构并提交。每个阶段必须通过以下适用门禁：

- 前端：格式化、Lint、类型检查、组件测试、关键流程 E2E 和可访问性检查。
- Java：单元测试、Web/Data Slice、模块依赖规则、Testcontainers 集成测试和状态机测试。
- API：OpenAPI 校验、客户端类型同步、消费者契约测试和错误结构测试。
- 数据：版本化 Migration、种子数据可重复执行、索引检查和备份恢复验证。
- Agent：确定性单元测试、工具权限测试、离线检索评测、回答忠实度、超时和降级测试。
- 安全：Secret、依赖和静态扫描，越权、水平权限和上传边界测试。
- 运行：容器健康检查、结构化日志、指标、Trace、告警规则和故障恢复演练。
- 完成声明：只有实际执行相应验证命令并检查输出后，才能声明功能完成或问题修复。

## 11. 开发阶段

### 阶段 0：需求与架构基线

产出角色与权限矩阵、领域词汇、核心用例、状态机、非功能需求、系统上下文、模块边界、ER 初稿、API 规范、威胁模型、ADR 和阶段验收标准。门禁是需求可测试、关键决策无歧义且不存在未归属的核心业务能力。

### 阶段 1：工程与环境基线

产出 Monorepo、双前端、Java 模块化单体骨架、数据库迁移、Compose `core`、统一代码规范和 CI。门禁是一条命令可启动基础环境，所有空骨架构建和检查通过。

### 阶段 2：体验、契约与 Mock

产出信息架构、设计系统、关键页面原型、OpenAPI、类型安全客户端、Agent 契约、Mock Agent 和演示种子数据。门禁是关键流程可以在 Mock 数据下点击演示，前后端契约测试通过。

### 阶段 3：消费者业务闭环

实现登录、商品、订单、物流、售后申请、咨询入口、工单进度和评价。门禁是 C 端关键 E2E、资源归属、幂等和异常路径测试通过。

### 阶段 4：客服与工单闭环

实现客服队列、实时会话、转人工、建单、分派、状态流转、SLA 和通知。门禁是状态机、并发接入、角色权限和跨部门流程测试通过。

### 阶段 5：知识库与工程增强

实现文档存储、RabbitMQ、Elasticsearch、异步任务、审计日志和数据看板。门禁是重试、死信、幂等、数据一致性和恢复测试通过。

### 阶段 6：真实 Agent 服务

实现 FastAPI Agent、意图识别、RAG、业务工具、自动建单建议、摘要、推荐回复和降级。门禁是离线评测、工具权限、引用、幻觉、超时和人工确认测试通过。

### 阶段 7：企业化加固

完成安全扫描、限流、可观测性、性能测试、备份恢复和故障演练。门禁是定义的 SLO、性能预算和安全基线全部有验证证据。

### 阶段 8：发布与作品包装

完成完整 Compose、演示数据、部署手册、Runbook、架构文档、演示脚本和简历量化指标。门禁是在全新环境按文档完成部署并通过最终验收清单。

## 13. 第 0 阶段的执行顺序

1. 将初稿拆为可验收的角色、业务能力和用例清单。
2. 建立领域词汇表和系统边界，消除“售后申请”“退款申请”“工单”“会话”等概念混用。
3. 定义会话、售后申请和工单状态机及非法状态迁移。
4. 定义 RBAC、部门、资源归属和租户上下文矩阵。
5. 建立模块依赖图、系统上下文图和关键时序图。
6. 设计核心实体、聚合边界、ER 初稿和数据保留规则。
7. 定义 API 约定、错误模型、分页、幂等、并发和事件规范。
8. 定义可度量的非功能需求、SLO、性能预算和容量假设。
9. 完成威胁模型、信任边界和首版安全控制清单。
10. 通过 ADR 固化技术选型和当前官方兼容版本，并形成阶段 1 的可执行实施计划。

## 14. 项目成功标准

- 无真实模型密钥时，Mock Agent 仍可完整演示消费者咨询到工单关闭的业务闭环。
- 接入真实 Agent 后，前端和核心 Java 业务不需要改变调用契约。
- 两个前端应用、Java 应用和基础中间件可由文档化命令构建、测试和启动。
- 关键业务规则、权限、状态迁移和并发冲突都有自动化测试证据。
- 全新环境可使用 Compose 和种子数据复现演示系统。
- 最终简历描述中的性能、质量和 AI 效果指标均来自可重复的测试或评测结果，而不是估算。

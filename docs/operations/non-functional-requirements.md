# 非功能需求与验收目标

这些指标用于当前 16 GB 开发笔记本和轻量虚拟机上的可重复验收，不构成生产容量或高可用承诺。所有结果必须记录命令、数据规模、环境和原始报告。

## 1. 性能

| ID | 目标 | 验证方式 |
|---|---|---|
| NFR-PERF-001 | 非 Agent 读取 API 在 50 个虚拟用户下 p95 小于 300 ms，错误率低于 1% | 固定种子数据下运行 k6 读取场景 |
| NFR-PERF-002 | 非 Agent 写入 API 在 20 个虚拟用户下 p95 小于 500 ms，错误率低于 1% | k6 使用独立幂等键和测试账号 |
| NFR-PERF-003 | 本地网络聊天消息持久化确认 p95 小于 500 ms | WebSocket/API 混合场景并核对 message sequence |
| NFR-PERF-004 | Agent 同步回答总超时为 15 秒 | 注入慢响应，验证降级和转人工 |
| NFR-PERF-005 | 列表接口默认 20、最大 100 条，不允许无界查询 | API 契约测试和代码审查 |

## 2. 可靠性与一致性

- 核心事务状态只以 MySQL 提交结果为准。
- MQ、搜索、通知和统计采用最终一致；重复事件不得产生重复业务结果。
- 工单、售后、会话和评价的冲突更新返回 409，不允许最后写入覆盖。
- WebSocket 断线后可通过持久化消息游标恢复。
- Agent、Elasticsearch 或通知不可用时，订单、售后和人工工单核心流程继续工作。
- 演示环境恢复点目标 RPO 为 24 小时，恢复时间目标 RTO 为 2 小时。
- Stage 7 必须完成一次数据库备份恢复和一次搜索索引重建演练。

## 3. 安全

| ID | 目标 | 验证证据 |
|---|---|---|
| NFR-SEC-001 | 跨租户和跨消费者资源访问成功数为零 | BOLA/IDOR 集成测试 |
| NFR-SEC-002 | Agent 不能绕过 Java 工具授权或直接访问数据库 | 网络策略、工具契约和负向测试 |
| NFR-SEC-003 | 仓库、镜像和日志不包含真实密钥 | Secret 扫描和日志检查 |
| NFR-SEC-004 | 上传文件执行类型、大小、哈希、归属和访问校验 | 上传边界测试 |
| NFR-SEC-005 | 权限、配置、工单和 Agent 工具调用可审计 | 审计完整性测试 |
| NFR-SEC-006 | 登录、消息、上传和 Agent 调用具有限流 | 429 契约和限流测试 |

## 4. 可观测性

- 每个 HTTP 请求、WebSocket 消息、异步事件和 Agent 调用携带 correlationId。
- 日志使用结构化格式，至少包含时间、级别、服务、环境、租户、主体类型、correlationId、业务错误码和耗时。
- 日志禁止记录密码、完整令牌、API Key、Cookie、上传文件正文和不必要的个人数据。
- 指标覆盖请求量、延迟、错误率、队列积压、死信、Agent 成功率/超时率、搜索命中、工单超时和通知失败。
- 健康检查区分进程存活与依赖就绪，避免依赖短暂故障导致无限重启。
- Stage 7 定义本地 SLO 看板和至少五条可操作告警。

## 5. 可维护性

- 四个应用独立构建，但在一个仓库中共享契约和质量门禁。
- Java 模块依赖由 Spring Modulith 验证或 ArchUnit 自动检查。
- OpenAPI 与前端生成客户端在 CI 中检查漂移。
- 数据库只通过 Flyway 演进，已应用迁移不可修改。
- 所有功能和修复遵循失败测试、最小实现、验证和重构循环。
- 完成声明必须附带当前提交上的构建、测试或文档门禁输出。

## 6. 可用性与可访问性

- customer-web 和 staff-web 的核心流程支持键盘操作、可见焦点、语义标签和错误文本。
- 桌面工作台目标宽度 1280 px 以上，同时保证 1024 px 可用；消费者端覆盖 360 px 至桌面宽度。
- 前端在 API 暂时不可用时显示可恢复错误，不清除用户尚未提交的表单。
- 关键操作提供明确确认、结果和可追踪编号。

## 7. 容量假设

- 演示数据：1 个租户、1 万消费者、10 万订单、100 万消息、10 万工单、1 万知识分块。
- 日常演示并发：50 个消费者会话、10 名员工、5 个并行 Agent 请求。
- 超出该范围前先重新压测和容量评估，不根据本地结果推断生产容量。

# 游戏数字商品商城与 AI 客服工单调度平台阶段性文档

## 文档用途

本文档用于记录项目每个阶段已经完成的工作、形成的关键决策、产出文件、验证结果和下一阶段入口。后续每完成一个阶段，都在本文档中追加对应阶段总结。

项目采用“作品级 MVP 先落地，再逐步演进到生产级”的路线。当前仓库作为主 Monorepo，规划承载四个应用：

- `frontend/customer-web`：消费者端前端。
- `frontend/staff-web`：客服与管理端前端。
- `backend/platform-api`：Java Spring Boot 模块化单体后端。
- `backend/agent-service`：Python Agent 服务，后期接入。

仓库一级目录整理规则见 `resources/docs/architecture/repository-layout.md`。根目录按前端、后端、资源三类组织；临时日志、构建产物和截图不保留在根目录。

## 业务域调整记录：从泛电商售后转向游戏数字商品平台

**调整日期：2026-06-25**

### 调整原因

原始方向是“AI 电商售后客服与工单调度平台”，但泛电商平台容易自然扩展成淘宝、拼多多、京东式多商户模式，业务边界会变成商家入驻、店铺治理、平台结算、商家客服和多租户运营。这会显著扩大第一版范围，不利于个人项目在有限时间内做出强闭环。

新的方向调整为“游戏数字商品商城与 AI 客服工单平台”。它仍保留电商交易、客服、Agent、工单、权限和审计这些工程价值，但业务对象更聚焦：

- 游戏、DLC、CDK、礼物码和数字权益。
- CDK 未收到、兑换失败、礼物领取失败、退款申请和账号异常。
- 客服负责沟通与升级。
- 管理员负责退款、补发和账号处置工单。
- 超级管理员负责角色模块配置和系统级能力。

### 保留内容

- Monorepo 四应用结构保持不变。
- `customer-web` 和 `staff-web` 两个前端应用保持不变。
- Java Spring Boot 模块化单体 + Python Agent 服务的架构保持不变。
- Windows 主机开发、Linux VM 部署中间件的拓扑保持不变。
- 阶段编号和推进节奏保持不变。
- Mock Agent 先行、真实 Agent 后置的策略保持不变。

### 修改内容

- 产品名展示从 `AICare Desk` 调整为 `GameCare Desk`。
- C 端从“售后服务中心”调整为“深色沉浸式游戏商城 + 订单/权益客服入口”。
- B 端从“客服与部门工单后台”调整为“客服/管理员/超级管理员角色权限后台”。
- 数据模型从“商品/物流/售后申请/部门处理”调整为“游戏商品/数字权益/CDK/礼物/退款审核/账号处置/模块权限”。
- 工单处理人从多部门人员收敛为管理员和超级管理员，降低 OA 化复杂度。

## 阶段 0：需求与架构基线

**阶段状态：已完成**

**总结日期：2026-06-24**

### 1. 阶段目标

阶段 0 的目标不是写业务代码，而是把项目的边界、目标、核心业务流程、模块拆分、数据模型、安全要求和后续开发阶段固化下来，避免后续进入工程初始化后频繁推翻基础设计。

本阶段确定项目定位为一个人开发、可写入简历、偏企业级、功能完整的 AI 电商售后客服与工单调度平台。第一版优先完成可演示、可测试的业务闭环，不追求真实高并发生产集群。

### 2. 已确认的核心决策

- 交付策略：先完成作品级 MVP，再根据需要扩展到生产级能力。
- 仓库形态：采用一个 GitHub Monorepo，承载两个前端、一个 Java 后端和一个 Python Agent 服务。
- 前端形态：`customer-web` 和 `staff-web` 独立开发、独立路由、独立构建，共享基础组件和类型。
- 后端形态：Java Spring Boot 模块化单体作为核心业务系统。
- Agent 形态：Python Agent 作为独立服务，最后开发；Java 先保留稳定 Agent 契约和 Mock 实现。
- 数据策略：第一版使用虚拟演示数据，不接入真实电商、支付、退款打款或物流平台。
- 部署策略：Windows 主机负责日常开发和调试，Linux 虚拟机负责 MySQL、Redis 等中间件与后续集成部署。
- 资源策略：考虑本机 16 GB 内存，虚拟机和 Docker Compose 采用分组 profile，日常只启动当前阶段必要组件。

### 4. 业务基线

阶段 0 将 MVP 业务闭环拆成 8 个核心用例：

- 消费者查看订单并发起咨询。
- Mock Agent 回答确定性订单问题。
- 会话转人工队列。
- 客服接入并回复。
- 消费者提交带凭证的质量类售后申请。
- 客服创建并分派工单。
- 部门专员处理工单。
- 消费者确认处理结果并评价。

对应验收标准已拆成 24 条可观察结果，编号为 `AC-001` 到 `AC-024`，并在 `resources/docs/stage-0-traceability.md` 中映射到用例、模块、接口或事件、数据实体、安全控制和计划测试层。

### 5. 架构基线

阶段 0 固化了四个主要系统边界：

- 浏览器前端不能直接访问 Agent 服务。
- Agent 服务不能直接访问 MySQL，也不能绕过 Java 后端执行业务写操作。
- Java 后端负责认证、授权、事务、审计、状态机和业务规则。
- Agent 只通过 Java 暴露的受控工具接口完成建议、摘要、问答和草稿生成。

Java 后端模块边界按业务能力拆分，包括 `identity`、`catalog`、`order`、`aftersales`、`conversation`、`ticket`、`knowledge`、`notification`、`analytics`、`agent-gateway`。

### 6. 安全与质量基线

阶段 0 明确了以下安全和质量要求：

- 所有核心资源必须校验租户、消费者归属、部门范围或分派关系。
- 工单、会话、售后申请必须使用显式状态机，非法状态迁移返回稳定业务错误。
- 重复提交、转人工、建单、评价等关键操作需要幂等控制。
- Agent 失败或超时时不能阻塞核心售后流程，必须降级并提供转人工入口。
- 文件上传需要校验类型、大小、魔数和对象隔离。
- 所有请求、事件、审计和 Agent 调用需要关联 ID。
- 最终简历和演示中使用的性能、质量和 AI 效果指标必须来自可重复验证结果。

## 阶段 2：体验、契约与 Mock

**阶段状态：进行中**

### 1. 当前阶段目标

阶段 2 的目标是把第 1 阶段的可运行骨架推进到可演示的体验和契约基线。当前优先做页面体验，后续继续补 OpenAPI、类型安全客户端、Java AgentGateway 契约、Mock Agent 和演示种子数据。

### 2. 页面设计方向

2026-06-25 根据业务域调整，重新使用 `ui-ux-pro-max` 进行游戏数字商品平台设计方向检索。新的页面方向从浅色企业售后工作台调整为深色、沉浸式、偏游戏商城和控制台风格：

- C 端参考 Steam、Epic Games Store、PlayStation Store 和 Xbox/Microsoft Store 的公开信息架构经验：顶部导航、搜索、游戏卡片、详情页、购物车/个人入口和客服入口。

设计说明已记录在 `resources/docs/design/stage-2-ui-direction.md`。

### 3. 消费者端原型

`frontend/customer-web` 已从原售后服务中心原型重做为游戏数字商品商城首页原型，当前包含：

- 顶部导航：分类、优惠、限时秒杀、订单、客服、公告。
- 顶部导航均预留跳转入口，其中分类和公告暂时不进入本阶段开发。

- 首屏右侧为过去 24h 销量榜。
- 推荐游戏下方为精选游戏列表。
- 点击推荐游戏或精选游戏卡片进入独立游戏详情页入口。
- 已移除首页内的 AI Support 模块，Agent 能力后续通过客服会话功能体现。
- 已移除首页内的分类条和个人资料条。

该页面当前仍使用静态演示数据，后续由 OpenAPI 和 Mock 数据接入。当前不是最终全量原型，游戏详情页、订单页、客服会话页、优惠页和限时秒杀页仍需要继续做路由级页面设计。
